# Gap analysis: risks and missing pieces

*A completeness-critic agent identified what the research missed. These are design inputs, not blockers.*

## 1. Vendor Terms of Use contradict the permissive robots.txt findings — ASUS ToS explicitly prohibits scraping. The findings judged ASUS/ASRock 'clean scrape targets' from robots.txt alone, but ASUS's Terms of Use (Section 1.8.4, verified-live this session) prohibits 'using any data mining, robots, or similar data gathering and/or extraction tools'. ASRock, MSI and Gigabyte have similar boilerplate. Separately, Geizhals content is protected by the EU sui generis database right (Directive 96/9/EC), which applies to extraction of substantial parts regardless of ToS or bot-blocking — relevant if Ivan is in the EU.

**Why it matters:** Risk posture changes depending on whether the catalog stays private or gets published. Private personal use at low rates is a de-facto non-issue (worst realistic outcome: IP ban), but if Ivan ever publishes the site or contributes scraped vendor data upstream to BuildCores, he's redistributing data collected in breach of contract. In the US, hiQ v. LinkedIn settled that public-page scraping isn't a CFAA violation, but breach-of-contract and copyright claims survive; in the EU, DSM Art. 4 TDM reservations (which TechPowerUp already invokes) can be asserted by any vendor. He should decide the private-vs-public question before building the pipeline, not after.

**Partial answer:** Verified live: https://www.asus.com/us/terms_of_use_notice_privacy_policy/official-site/ Section 1.8.4 bans data-mining/robots. Practical mitigation: keep scraped vendor data private, publish only original normalized facts (spec facts themselves are not copyrightable in the US — Feist), attribute BuildCores/Icecat for redistributable layers, and rate-limit to indistinguishable-from-human levels.

## 2. BIOS history is lossy at the source — vendors silently pull BIOS releases (especially betas and bricked versions), so the firmware-history differentiator cannot be fully backfilled later. Nothing in the findings addresses that the ASUS/ASRock lists Ivan scrapes today are already incomplete: MSI routinely removes faulty/beta BIOSes, ASUS drops beta versions after stable release, and the Reous sheet exists precisely because vendors delete releases. Every day Ivan doesn't run the poller, history evaporates.

**Why it matters:** This is the core product differentiator. If the value proposition is 'BIOS version history retailers never surface', an append-only capture model (never delete a version just because the vendor did; flag it 'pulled' instead — a pulled BIOS is itself high-value signal) must be designed in from day one, and the poller should be among the first things built, even before the facet UI.

**Partial answer:** Backfill is partially possible: the Wayback Machine CDX API works and even holds archived ASUS GetPDBIOS JSON responses (verified live this session via http://web.archive.org/cdx/search/cdx?url=www.asus.com/support/api/product.asmx/GetPDBIOS*&output=json — returned 200 captures from 2023-2025). Old ASRock bios.html and Gigabyte support pages are similarly archived; a CDX-driven backfill script can reconstruct multi-year history per board. Also: soggi.org/motherboards/msi.htm archives MSI BIOSes including pulled betas.

## 3. No product-identity strategy across sources — the findings recommend GTIN as canonical key, but one physical board has MANY GTINs and IDs: regional/packaging EAN vs UPC variants, vendor internal part numbers (ASUS 90MB-prefixed SKUs — the Icecat sample itself is keyed 90MB17A0-M0EAY0), Gigabyte revision-embedded URLs where 'rev 1.0/1.1' share one page but 'V2' is a separate product, BuildCores UUIDs, Icecat product IDs, and regional model-name variants (same board sold as different names in different markets, D4/DDR4 suffixes, WIFI vs non-WIFI SKUs sharing a BIOS). Nothing specifies the entity-resolution/alias layer that joins BuildCores ↔ Icecat ↔ vendor pages ↔ linuxhw DMI strings (which use DMI names like 'MS-7C02', not marketing names).

**Why it matters:** Every enrichment join in the recommended pipeline (BuildCores seed + Icecat GTIN + vendor scrape + DMI corroboration) silently depends on this matching working. Without an identity table (product → alias records typed by source: gtin_ean, gtin_upc, vendor_sku, dmi_name, buildcores_id, icecat_id, marketing_name+region), Ivan will get duplicate boards and mismatched BIOS histories — the exact failure mode that makes catalog sites untrustworthy. The DMI-name problem is the worst: MSI DMI strings are 'MS-7Dxx' codes needing a lookup table that doesn't exist as open data.

## 4. PCIe/M.2 lane-sharing data lives almost exclusively in PDF manuals and spec-page footnotes, and no finding covers PDF ingestion. The schema finding correctly adds a shared_with field to pcie_slot, but no identified source populates it: BuildCores doesn't have it, Icecat doesn't, and vendor HTML spec tables carry it only as asterisked free-text footnotes ('*When M.2_2 is occupied, PCIe x16_2 runs x2'). The authoritative source is the board manual PDF (block diagrams are often images inside the PDF, requiring OCR or manual entry). BuildCores' README explicitly lists 'product PDFs (motherboard manuals)' as unmet help-wanted work — confirming nobody has solved this.

**Why it matters:** Ivan's own example query — '3+ PCIe slots' — is exactly the query lane-sharing invalidates: a board can advertise 3 physical slots where populating any M.2 disables one. Without sharing data the flagship facet returns wrong answers on precisely the mATX/ITX boards he cares about, where sharing is most aggressive. He should budget for a semi-manual workflow (PDF download + LLM-assisted extraction + human confirm flag per board) rather than assuming full automation, and the schema needs a 'sharing_verified' provenance bit.

## 5. No staleness/refresh/change-detection design beyond the ASRock BIOS feed. Missing: (a) new-product discovery — how does the system learn a new board launched? (vendor sitemap.xml files and category listing pages are the frontier; only ASRock's /mb/index.asp was noted); (b) spec pages change silently — vendors correct errors and even alter specs post-launch (memory OC limits get raised with BIOS updates), so scrapes need content-hash snapshots and diffing, not overwrite-in-place; (c) CPU-support/QVL lists are themselves time-varying documents (since-BIOS entries appear, beta rows vanish) and need snapshotting too; (d) EOL delisting — vendors remove product pages entirely, so raw HTML payloads must be archived at scrape time (the schema's source table stores raw payload, but no retention/replay policy is stated); (e) polling cadence per source class is unspecified except one suggestion buried in the BIOS finding.

**Why it matters:** A catalog that's wrong is worse than no catalog — stale max-memory specs and vanished EOL boards are the two most common failure modes of exactly the retail sites Ivan is frustrated with. Append-only versioned ingestion (scrape_snapshot table keyed by URL+content_hash+date) costs almost nothing at this scale and makes every later correction and 'when did this spec change' question answerable.

## 6. Gigabyte-style revision chaos is worse than the findings model: revisions share pages unpredictably, and hardware differs within a shared page. Verified live via Gigabyte support pages: 'B650 EAGLE AX (rev. 1.0/1.1)' is ONE page covering two revisions whose 1.0 uses AMD WiFi and 1.1 uses Intel WiFi (separate driver downloads per revision on the same page), while 'B650 GAMING X AX V2' is a separate product from 'B650 GAMING X AX'. Meanwhile B550M DS3H spans rev 1.0/1.1/1.2/1.3 on one page. The proposed product→board_revision schema is right, but the scraper design must handle one-URL-many-revisions and disambiguate per-revision downloads by parsing the rev labels attached to each download row — and sometimes the spec table itself doesn't say which revision a line applies to.

**Why it matters:** If the scraper naively maps one URL to one product, Gigabyte data will be wrong at the revision level — and BIOS files are per-revision (flashing rev-1.0 BIOS on rev-1.1 can brick), so the firmware-history feature has a correctness/safety obligation here, not just a modeling nicety.

**Partial answer:** Sources verified: https://www.gigabyte.com/Motherboard/B650-EAGLE-AX-rev-10-11/support (per-rev WiFi drivers), https://www.gigabyte.com/Motherboard/B550M-DS3H-rev-10-11-12-13. Physical boards print the revision on the PCB ('REV: 1.1' silkscreen) — worth storing as a 'how to identify' note per revision.

## 7. BuildCores local-corrections-vs-upstream-sync conflict is unaddressed. The seeding recommendation says 'periodically git pull upstream', and the data-quality finding says 'fix suspect records' — but doing both means either losing local fixes on every pull or forking permanently. There's no overlay design: upstream data must be immutable in Ivan's DB, with local corrections stored as a separate patch/override layer carrying provenance and precedence (manual > vendor-scrape > BuildCores), re-applied after each sync.

**Why it matters:** This is the classic downstream-catalog trap; without it the first git pull after a month of manual QA destroys the QA. The same precedence machinery is needed anyway to merge Icecat vs vendor-page disagreements, so it should be a core design element, not an afterthought. Bonus: a clean override layer makes upstreaming fixes to BuildCores (which solicits contributions) trivial to generate as PRs.

## 8. Cross-category compatibility relations — the thing that will actually stress the schema when generalizing — are absent. The findings treat 'other categories' as 'more attribute rows + new child tables', but the hard part is relations BETWEEN products: cooler↔socket compatibility lists, case↔form-factor/clearance (GPU length vs case max-length is a range-vs-scalar join), PSU↔GPU connector matching (12V-2x6 vs 12VHPWR revisions), RAM QVL (board×kit matrix, per-BIOS-version!), and CPU↔board (the since-BIOS support lists already found, which are literally a three-way relation product×product×bios_release). None of the proposed schemas (BuildCores, Icecat, the synthesized one) model product-to-product edges.

**Why it matters:** Ivan's stated trajectory is 'starts with motherboards but should generalize'. If v1's schema has no concept of a compatibility edge (subject_product, object_product_or_class, constraint_type, evidence_source, as_of_bios), retrofitting it means reworking the core. The QVL data he's already planning to scrape for the BIOS feature IS this edge type — recognizing that early gets cross-category compatibility nearly free.

## 9. Facet semantics for 'conditional' spec values are undefined: memory speed 'DDR5 5600(OC)' vs JEDEC native, max-memory figures that changed post-launch via BIOS (48GB DIMM support added by AGESA updates), USB port counts that mix chipset-native and hub-attached, and 'shared' audio jacks. The vocabulary-normalization finding handles synonyms and units but not conditionality — a spec whose value depends on configuration (OC enabled, DIMMs per channel, BIOS version).

**Why it matters:** These produce the misleading filters Ivan is building this tool to escape. Each numeric facet needs an explicit semantic: store max_memory_speed_native and max_memory_speed_oc as separate attributes (ETIM-style), and date/BIOS-qualify values that changed over time — which dovetails with the BIOS-history table he's building anyway (e.g., 'supports 256GB since BIOS 1.2.0.2').

## 10. Redistribution/archiving legality of BIOS binaries themselves. Because vendors delete releases (gap above), a complete history implies archiving the actual firmware files — but BIOS images are copyrighted vendor software plus licensed components (AMD AGESA, Intel reference code); redistributing them is what keeps sites like soggi.org and the old Win-Raid archives in a legal gray zone. Storing metadata (version/date/changelog/SHA256/original URL) is safe; hosting the files publicly is not.

**Why it matters:** Decides storage architecture now: keep a private binary archive (fine for personal use, enables later microcode/AGESA extraction with MCExtractor/UBU per the Win-Raid methodology finding) but only ever publish metadata + SHA256 + link-to-vendor. SHA256s are also the join key to identify re-hosted files later if a vendor link dies.

## 11. As-documented vs as-shipped component variance within a single revision: vendors ship different NIC/WiFi/audio parts under one model+revision when supply changes ('specifications may vary', silent Realtek↔Intel WiFi module swaps are documented on Gigabyte's own per-rev driver pages, and SSD-style silent controller swaps have hit motherboard NICs too). A 'Intel 2.5GbE only' facet built from spec-page text can be wrong for the unit actually purchased.

**Why it matters:** Directly undermines the '2.5GbE controller' facet trust. Mitigation is already in the findings but unconnected: linuxhw/DMI + LsPCI dumps show what hardware actually shipped in the wild — cross-referencing observed PCI IDs against documented controllers per board would flag variance cases. The schema needs a confidence/observed flag per component claim, not just per record.

## 12. ASUS undocumented API fragility and enumeration: GetPDBIOS requires the exact internal model string, and no finding covers how to enumerate all motherboard model names/pdids sitewide (the ASRock finding got a free crawl frontier from /mb/index.asp; ASUS's equivalent — the product-list JSON inside window.__NUXT__ on series listing pages, or a products API — was not identified). Also unaddressed: website=us vs website=global can return different BIOS lists (regional builds), and an undocumented frontend API can add signing/aktime params or Akamai protection any day (ASUS already fronts other subdomains with bot management).

**Why it matters:** The daily-poll architecture leans hardest on this one endpoint. It needs (a) a model-name enumeration step, (b) contract tests that alert when response shape changes, and (c) the Wayback CDX backfill (verified above) as the disaster-recovery path if ASUS locks it down.

## 13. No Intel-side equivalent of the Reous AGESA tracker was identified, and Intel platform firmware has an extra dimension the findings skip: ME (Management Engine) firmware versions, which ship both inside BIOS updates and as separate 'ME Firmware' downloads on vendor support pages, plus Intel's CSME security advisories that make ME version a real filter/health signal. The microcode finding covers CPU microcode only.

**Why it matters:** If BIOS-history is the differentiator, Intel boards deserve parity with the AMD/AGESA treatment: parse 'Update ME to 16.x.x' strings from changelogs and scrape the ME-firmware download rows (same vendor support pages, same scraper). Otherwise half the catalog has second-class firmware data.
